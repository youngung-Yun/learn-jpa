# JPA, Java Bean Validation, Flyway 학습

## JPA

기본적으로 서브 테이블(FK를 가진 테이블)의 엔티티 쪽에서 메인 테이블(외래 키의 출처 테이블) 쪽으로 연관관계를 가진다.
```
member
- id(PK)
- name
- email

order
id(PK)
member_id(FK)
```
테이블이 이 구조일 때 `order` 테이블이 서브 테이블이며 `member` 테이블이 메인 테이블이다.

또한 하나의 `member`가 여러 개의 `order`를 가질 수 있으므로 1:N 관계이다.

서브 테이블의 경우는 메인 테이블의 엔티티를 참조하면서, `@JoinColumn`으로 테이블 상에서 내가 가진 FK의 정보를 넣는다.

양방향 연관관계의 경우, 메인 테이블의 엔티티 또한 서브 테이블의 엔티티를 참조로 갖는다.

이 경우 `@OneToOne` 또는 `@OneToMany`의 `mapped` 속성에, 서브 테이블 엔티티가 나의 참조를 가지고 있는 `필드명`을 넣는다.

### 지연 로딩(Lazy Loading)
- 기본적으로 엔티티 조회 시 연관된 엔티티까지 한 번에 조회하여 가져온다.
- 지연 로딩을 사용하면 조회 전까지 프록시를 가지고 있으며, 실제로 연관된 엔티티를 참조할 때 까지 조회가 지연된다.
- `FetchType.LAZY`

## Cascade
- cascade를 설정하면 해당 엔티티의 작업을 연관된 서브 엔티티에도 전파

## orphanRemoval
- 메인 엔티티가 없는 서브 엔티티는 필요 없으므로 삭제함

## N + 1문제
- N개의 엔티티를 1번의 쿼리로 조회한 후, 각 엔티티에 대해 연관관계인 엔티티를 쿼리 1번씩으로 조회, 총 1 + N개의 쿼리가 발생하는 현상
### 해결 방안
1. `fetch join`
   - JPQL에 `join`대신 `fetch join`을 선언하여 조회시 join문으로 한 번에 가져옴
   - 직접 JPQL 작성해서 조회해야 함
2. `@EntityGraph`
   - 엔티티 그래프에 포함된 객체를 join문으로 한 번에 가져오는 방식
3. `@BatchSize`
   - size 만큼의 연관관계인 엔티티를 in문으로 여러 개 가져오는 방식
   - 페이징과 함께 사용하면 문제 발생

## Java Bean Validation
- 엔드포인트 요청 바디의 DTO 필드를 자동으로 검증해줌
- 검증 실패시 예외 발생

## Flyway
- DB 스키마 파일 버전 관리 도구
- 의존성 추가시 스프링시 자동으로 실행하여 DB에 스키마 생성함
- 스키마 파일은 증분 구조로, 기존 버전 파일을 수정하지 않고 새로운 버전의 파일에 수정 사항을 넣음.
